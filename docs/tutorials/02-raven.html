Raven: RStudio Server & Reproducible Analysis

Run an interactive analysis on the lab's shared server and push the result back to GitHub. Raven is the middle tier — more power than your laptop, gentler than the HPC cluster.

~1 hourRaven RStudio ServerFinish Tutorial 1 first
Step 1

Why a server? (server vs. laptop)

Raven is a computer that lives in the department and runs RStudio in your browser. You use its CPU, memory, and disk — not your laptop's.

  • More resources: bigger jobs than your laptop can handle, without tying up your machine.
  • Shared environment: everyone has the same software installed, so "works on my machine" problems shrink.
  • It's still just files + Git: the same clone → commit → push loop from Tutorial 1 works here.
  • Shared, not private: your home directory is yours, but the machine is communal — be tidy and don't hog resources.
Handbook: Raven RStudio Server and Raven Conda are your references for this tutorial.
Step 2

Log into Raven

Open http://raven.fish.washington.edu:8787 in your browser and sign in with the credentials the instructor provides. You'll land in a full RStudio IDE — Console, Files pane, Terminal tab, and all.

Access note: Raven may require being on the campus network or UW VPN (Husky OnNet). If the page won't load off campus, connect to the VPN first.

Open the Terminal tab inside RStudio (next to Console) — this is a real shell on Raven, where your Tutorial 1 commands work.

whoami
pwd
/home/your-username
Step 3

Clone your repo on the server

Your GitHub repo is the bridge between machines: clone it on Raven exactly like you did on your laptop.

mkdir -p ~/GitHub
cd ~/GitHub
git clone https://github.com/course-fish546-2026/your-repo.git
cd your-repo

Set your Git identity on Raven the first time (so commits are attributed to you):

git config --global user.name  "Your Name"
git config --global user.email "you@uw.edu"
Now the same project exists in three places by the end of the setup: your laptop, Raven, and GitHub. GitHub is the source of truth; you git pull to sync a machine before working and git push when done.
Step 4

Environments & Conda

Reproducibility means pinning which software versions you used. On Raven, Conda manages that.

List available environments and activate one:

conda env list
conda activate <env-name>

Check what's installed — e.g. R and a common tool:

R --version
which samtools

If you need a package that isn't there, note it in an issue rather than installing globally — the Raven Conda guide shows the right way to make your own environment.

Step 5

Run something and push the result

Prove the round trip: compute on Raven, write to output/, commit, push, and see it on GitHub.

In the RStudio Console, run a tiny analysis that writes a figure to your repo's output/ directory:

setwd("~/GitHub/your-repo")
dir.create("output", showWarnings = FALSE)
png("output/hello-raven.png", width = 700, height = 450)
hist(rnorm(1000), col = "#0b7285",
     main = "Hello from Raven", xlab = "value")
dev.off()

Then, in the Terminal tab, commit and push:

git add output/hello-raven.png
git commit -m "Add first Raven output figure"
git push
Refresh your repo on GitHub — the PNG should be there. That's a complete reproducible cycle on the server.
Step 6

Data hygiene on a shared server

  • Don't commit big data. Sequence files and large outputs don't belong in Git. Add them to .gitignore and keep them in data/ on the server.
  • Record provenance. When you bring in raw data, store a checksum/manifest so you can prove it didn't change: 
    md5sum data/raw/*.fastq.gz > data/raw/md5.txt
  • Clean up. Delete scratch files you no longer need — the disk is shared.
Handbook: Data Management covers metadata, backups, and archiving — the standard for the quarter.
