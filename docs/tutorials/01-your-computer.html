Your Computer: Shell & Git/GitHub

Get a working local setup and learn the everyday version-control loop. This is the foundation for everything else in the course — do it first.

~1 hourYour laptop/desktopNo prior experience needed
Step 1

Install the tools

You need three things: a terminal, Git, and a GitHub account (plus the GitHub Desktop app if you prefer clicking to typing).

GitHub account

Create a free account at github.com/join if you don't have one. Use a name you'd be comfortable with a future employer seeing. Then tell the instructor your username so you can be added to the course organization.

Git + GitHub Desktop

Install GitHub Desktop — it bundles Git and gives you a friendly interface for the daily loop. On macOS you already have a terminal (Terminal.app); on Windows, GitHub Desktop installs Git Bash, which you'll use as your terminal.

Verify Git works

Open your terminal and run:

git --version
git version 2.43.0   # any recent version is fine
Handbook: This mirrors the Computing Best Practices page and the lab's Git & GitHub tutorial. Keep them open.
Step 2

Where your repositories live

A little discipline here prevents a lot of pain later.

  • Create one dedicated folder for all GitHub repos. On a personal computer, ~/GitHub/ is the convention.
  • Do not put repositories inside folders that auto-sync with iCloud Drive, Dropbox, Google Drive, or OneDrive. Sync services corrupt Git state.
  • Once a repo is cloned, don't casually move or rename it — code and settings depend on its location.
mkdir -p ~/GitHub
cd ~/GitHub
Why it matters: "My code can't find the file" is the most common early problem, and it's almost always because files moved or live in a synced folder. Pick a home and leave it there.
Step 3

Shell basics

You'll live in the terminal all quarter. Five commands get you 80% of the way.

CommandDoes
pwdprint working directory — "where am I?"
lslist files here (ls -l for detail, ls -a to show hidden)
cd <dir>change directory (cd .. goes up one)
mkdir <name>make a directory
cat <file>print a file's contents

Try it — every path is a real location on disk:

cd ~/GitHub
pwd
/Users/you/GitHub
ls -la
Go deeper: The handbook's bash tutorial is a full hands-on walkthrough. If the shell is brand new to you, work through it now.
Step 4

Project structure

Every project in this course uses the same skeleton. A README.md in each directory says what's inside.

Project-01/
├── README.md
├── code/
│   └── README.md
├── data/
│   ├── raw/
│   │   └── README.md
│   └── README.md
└── output/
    └── README.md

The rules that make this reproducible:

  • code reads from data and writes to output — never the other way around.
  • Raw data in data/raw/ is read-only; you never edit it by hand.
  • Number files so the order of operations is obvious: 01-QC.Rmd, 02-DESeq2.Rmd.
  • Anyone should be able to clone your repo and understand it from the READMEs.
Handbook: This is straight from Data Management and Computing Best Practices — the standard you'll be assessed against.
Step 5

The daily loop: clone → edit → commit → push

This four-beat loop is how work moves from your machine to GitHub. You'll do it dozens of times a week.

Create your course repo

On GitHub, create a new repository in the course org named for yourself (e.g. your-name-546), with a README. Then clone it — via GitHub Desktop (File → Clone repository) or the command line:

cd ~/GitHub
git clone https://github.com/course-fish546-2026/your-repo.git
cd your-repo

Build the skeleton

mkdir -p code data/raw output
printf "# code\nScripts for this project.\n" > code/README.md

Commit and push

A commit is a labeled snapshot. Push sends it to GitHub.

git add .
git commit -m "Add project directory skeleton"
git push

In GitHub Desktop this is the same three beats: check the changed files → write a summary → Commit → Push origin.

Good commit messages describe what changed and why in the imperative: "Add QC script", not "stuff" or "update". Your future self reads these.
