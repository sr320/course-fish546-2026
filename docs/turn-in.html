Turn In Work

Pull-request submission and automatic pre-checks

The idea

You turn in question sets and assignments in one place: the assignments repository. You are a collaborator on it, so you clone it, do your work in your own folder, push a branch, and open a pull request. A bot pre-checks the pull request and comments with a report; the instructor reviews the science and merges.

Everyone uses the same loop:

  1. Clone the assignments repo (once) and make your folder: submissions/<your-username>/.
  2. Start a branch and do the week’s work under submissions/<your-username>/weekNN/.
  3. Commit your files, including a rendered report and a submission.yml manifest.
  4. Push the branch and open a pull request into main.
  5. Read the automatic pre-check comment and fix anything it flags.

The pull request is the receipt. The branch freezes the submitted version. The manifest tells the checker what evidence belongs to the submission.

Where your work lives

All of your work for the whole term lives under one folder, one subfolder per week:

submissions/
  <your-username>/
    week02/
      analysis.qmd            # or answers.md for a question set
      analysis.html           # you render this yourself
      output/read-counts.csv
      output/qc-summary.png
      submission.yml          # the manifest

Only edit files inside submissions/<your-username>/. The bot fails any pull request that touches another student’s folder or the assignment prompts.

What gets checked automatically

The automatic check is not meant to replace instructor judgment. It checks the things software is good at checking:

  • Your pull request only touches your own folder.
  • A submission.yml manifest is present in the week folder you changed.
  • The manifest identifies the week, type, title, and submitted files.
  • The source file and rendered report exist and are non-empty.
  • Output tables and figures listed in the manifest exist and are non-empty.
  • Raw data are not committed under any data/raw/ directory.

The instructor still evaluates interpretation, biological reasoning, caveats, and scientific quality. The bot does not run or render your code — you commit your own rendered .html.

The manifest

Copy templates/submission.yml into your week folder and fill it in. All paths are relative to the folder that holds submission.yml (your week folder).

Question set example:

type: question-set
week: "02"
title: "Raw reads, FASTQ, QC, and trimming"
source: answers.md
rendered: answers.html
questions_answered: 5
project_connection: true

Assignment example:

type: assignment
week: "02"
title: "FASTQ Quality Control"
source: analysis.qmd
rendered: analysis.html
outputs:
  - output/read-counts.csv
  - output/qc-summary.png
uses_raw_data: true
raw_data_committed: false
project_connection: true

The automatic report

After you open or update the pull request, GitHub Actions posts a report as a comment (updated in place each time you push) like:

Automatic pre-check: warn

Passed
- All changed files are under your folder `submissions/<you>/`.
- Manifest type is `assignment`.
- Rendered file found.
- Output file found.

Warnings
- Manifest does not confirm a project connection.

Needs human review
- Biological interpretation and caveats.

The pull request also gets a label: auto:pass, auto:warn, or auto:fail, and the pull-request check turns red on fail. If it fails, fix the flagged items, commit, and push to the same branch — the pre-check runs again automatically.

Why this is better than uploads or issues

This system assesses the same habits the course is trying to teach:

  • Version-controlled work.
  • Reproducible project structure.
  • Clear evidence trails.
  • Precise submission versions (the branch and its commits).
  • Fast feedback before grading.
  • Human review reserved for scientific judgment.

It also lets the instructor see cohort-level progress: open pull requests are outstanding submissions, and auto:* labels triage them at a glance.

Weekly research progress

Weekly project progress is not a pull request. It is a structured project-management checkpoint, submitted through a Weekly Research Progress issue on the course repository.

Each progress issue asks for:

  • Week number.
  • Course repository URL.
  • Project issue URL.
  • Current commit SHA or GitHub compare URL.
  • What you accomplished this week.
  • Evidence links to commits, outputs, reports, figures, Issues, or PRs.
  • Whether last week’s goals were met.
  • Goals for next week.
  • Self-rating: On track, Some risk, or Blocked.
  • Blockers or help needs.

The automatic progress check comments on the issue and applies labels:

Label Meaning
progress:on-track The weekly record is complete and the student reports forward motion.
progress:watch The record is incomplete, thin, risky, or needs instructor attention.
progress:blocked The student reports being blocked and should have a linked Help Request or Blocker/Bug issue.

The progress checker looks for completeness and evidence, not whether the science is “good enough.” The instructor uses the report to review trajectory, intervene early, and assess weekly project-progress credit.
