Hyak / Klone: HPC with SLURM & Containers

Hyak (the Klone cluster) is where the heavy compute happens — RNA-seq, eDNA metabarcoding, big alignments. It has the steepest learning curve of the three platforms, so this tutorial goes slow and explains the why. Your only goal today: submit one trivial job and read its output.

Step 1

How a cluster works (read this before touching anything)

A cluster is not one big computer you log into and run things on. Understanding this one distinction prevents the most common beginner mistake.

Login node vs. compute nodes

  • When you ssh in, you land on a login node. It's shared by everyone and is only for light tasks: editing files, moving data, submitting jobs.
  • The real work runs on compute nodes, which you never touch directly. You ask the scheduler for them.
  • Never run analysis on the login node. It slows the cluster for everyone and admins will notice. This is the #1 rule.

The scheduler (SLURM)

You write a small script describing what you want (how many CPUs, how much memory, how long, which account) and hand it to SLURM with sbatch. SLURM finds a compute node, runs your job there, and writes the output to a file. You wait, then read the file.

Mental model: the login node is the front desk; SLURM is the host who seats you; the compute node is your table. You order at the desk, but you don't cook in the lobby.

Step 2

SSH into Klone

From your laptop's terminal (or Raven's), connect with your UW NetID:

ssh <UWNetID>@klone.hyak.uw.edu

You'll enter your UW password and approve a Duo two-factor prompt. On success you land on a login node:

[your-netid@klone-login01 ~]$
First-time access: you must be added to the lab's Hyak allocation (the srlab account) before jobs will run. If ssh is refused or you have no allocation, open a Help Request issue with the platform set to Hyak — this is expected and easy to fix.

Prefer a GUI? Hyak On-Demand gives you a browser dashboard (including RStudio on Klone) — a gentler on-ramp covered in the handbook's Klone RStudio guide.

Step 3

Where files go on Klone

Klone has several storage areas with very different rules. Put things in the wrong place and they get purged or you run out of space.

LocationUse forNote
/mmfs1/home/<netid>small config, scriptssmall quota — not for data
/gscratch/srlab/lab data & analysesthe lab's shared working space
/gscratch/scrubbed/large temporary workauto-deleted after a set time — never permanent

Make yourself a working directory in the lab space:

cd /gscratch/srlab/
mkdir -p <netid>/fish546-setup
cd <netid>/fish546-setup

Step 4

Moving data on and off

Small files — run this from your laptop (not from Klone):

# copy a local file up to Klone
scp myfile.txt <netid>@klone.hyak.uw.edu:/gscratch/srlab/<netid>/fish546-setup/

# sync a whole directory (resumable, skips unchanged files)
rsync -avP mydir/ <netid>@klone.hyak.uw.edu:/gscratch/srlab/<netid>/mydir/

Large datasets: use Globus (a managed transfer service) rather than scp — it's faster and recovers from interruptions.

Handbook: File Transfers covers scp, rsync, and Globus in detail.
Step 5

Software: modules & containers

You don't install bioinformatics tools system-wide on Klone. The lab runs them from Apptainer containers — a single file holding a whole reproducible software stack.

Load the container runtime with a module:

module load apptainer

Then run a command inside the lab container. The container file lives in /gscratch/srlab/containers/:

apptainer exec \
  --home $PWD \
  --bind /mmfs1/home/ \
  --bind /gscratch/ \
  /gscratch/srlab/containers/srlab-bioinformatics-container-<hash>.sif \
  blastn -version

The --bind flags make cluster storage visible inside the container. You'll reuse this exact pattern for every real tool this quarter.

Handbook: Using containers on Klone explains Apptainer, the current container hash, and binding.
Step 6 · the main event

Your first SLURM job

A SLURM job is a plain text script: a header of #SBATCH lines telling the scheduler what you need, then the commands to run. Create this file with nano in your working directory.

Create hello-slurm.sh:

#!/bin/bash
## --- SLURM header: what to ask the scheduler for ---
#SBATCH --job-name=hello-546
#SBATCH --account=srlab
#SBATCH --partition=ckpt
#SBATCH --nodes=1
#SBATCH --ntasks=1
#SBATCH --time=00:05:00
#SBATCH --mem=1G
#SBATCH --output=slurm-%j.out

## --- the actual work ---
echo "Hello from Klone!"
echo "Running on host: $(hostname)"
echo "Date: $(date)"
sleep 20
echo "Job finished."

Submit it to the scheduler:

sbatch hello-slurm.sh
Submitted batch job 1234567

That number is your job ID — you'll need it below.

The header matters: --account and --partition must match an allocation you belong to (the lab uses srlab). Ask too much --time or --mem and you wait longer in the queue; ask too little and the job is killed mid-run. Start small.
Step 7

Watch it, then read the output

Check the queue for your job (it may be PD pending, then R running):

squeue -u $USER

Need to cancel one? Use the job ID:

scancel 1234567

When it's gone from the queue, it finished. SLURM wrote everything your script printed to slurm-<jobid>.out. Read it:

cat slurm-1234567.out
Hello from Klone!
Running on host: n3148
Date: ...
Job finished.

That output file — and the fact it ran on a compute node (n3148), not the login node — is proof your job worked.

Step 8

Getting unstuck (you will need this)

Everyone hits walls on HPC — it's normal, not a sign you're doing it wrong. Here's how to diagnose and how to ask for help well.

SymptomLikely cause / fix
Job stuck in PD (pending) a long timeCluster is busy, or you asked for a lot of resources. Reduce --mem/--time, or wait.
Job vanishes instantly, tiny .outHeader error — wrong --account/--partition, or a typo. Read the top of the .out file.
command not foundYou're not inside the container. Wrap the command in apptainer exec ... (Step 5).
Killed with an out-of-memory messageRaise --mem and resubmit.
Permission denied on your scriptMake it executable: chmod +x yourscript.sh.
Asking for help the right way gets you unstuck fastest. Open a Help Request issue in your repo (platform = Hyak) with: what you were trying to do, the exact command, the full error text, and your job ID. This is the same skill you'll use all quarter — vague "it doesn't work" gets slow answers; a precise report gets fast ones.
